# Task 1.2 — Describe Each Component

## Data Sources

### Historical Batch File

The historical batch file contains past ecommerce transaction data stored in an Excel file (Online Retail dataset). It serves as the initial dataset used for analytics, reporting, and machine learning model training.

The dataset includes fields such as:

- InvoiceNo
- StockCode
- Quantity
- UnitPrice
- CustomerID
- InvoiceDate

The input format is `.xlsx`, and during ingestion it is converted into a structured format suitable for data processing.

---

### Live Transaction Stream

The live transaction stream represents real-time purchase events generated by customers on the ecommerce platform.

Transactions arrive continuously as individual records instead of batch files. Each event contains the same fields as the historical dataset and is processed immediately.

Streaming technologies such as APIs or event brokers deliver these events to the pipeline.

---

# Ingestion Layer

## Batch Ingestion

Batch ingestion processes historical datasets and loads them into the pipeline.

Tools that can perform this process include:

- Python ETL scripts
- Apache Airflow
- Apache Spark

Input: Historical dataset file  
Output: Structured data written into the raw storage layer.

---

## Stream Ingestion

Stream ingestion captures real-time transactions as they occur.

Common technologies include:

- Apache Kafka
- Streaming APIs

Input: Individual transaction events  
Output: Continuous stream of raw records stored in the landing zone.

---

# Landing Zone (Raw Storage)

The landing zone stores data exactly as it arrives from ingestion without any transformation.

Purpose:

- Preserve original raw data
- Enable reproducibility
- Provide an audit trail

Data is typically stored in a **data lake** using columnar formats such as **Parquet**, often partitioned by ingestion date.

---

# Validation Layer

The validation layer checks incoming records to ensure they meet structural and logical requirements.

Validation includes:

- Schema validation
- Value range checks
- Business rule validation

Tools such as **Great Expectations** or **Python validation scripts** can perform these checks.

Output:

- Valid records → transformation stage  
- Invalid records → quarantine storage

---

# Quarantine / Dead Letter Storage

The quarantine area stores records that fail validation checks.

Examples of errors:

- Missing values
- Invalid formats
- Business rule violations

These records are stored separately so corrupted data does not enter downstream systems.

Engineers or analysts can review and fix these records before reprocessing.

---

# Transformation Layer

The transformation layer processes validated data to clean and enrich it.

Typical operations include:

- Removing duplicates
- Standardizing formats
- Creating derived fields
- Generating aggregated metrics

Tools used may include:

- Apache Spark
- Pandas
- SQL transformations

Output: Structured datasets ready for analytics and machine learning.

---

# Clean Storage Layer

The clean storage layer stores curated datasets that are ready for analysis.

Characteristics:

- Data already validated and transformed
- Optimized for analytical queries

Formats commonly used:

- Parquet
- Delta Lake

---

# Feature Store

The feature store contains machine-learning-ready features derived from transaction data.

Example features:

- purchase frequency
- customer recency
- average order value

Storing features separately ensures consistency between training and prediction stages.

---

# Consumer Interfaces

## BI Dashboard

Business intelligence dashboards allow analysts and decision-makers to explore data visually.

Tools include:

- Tableau
- Power BI

Dashboards provide insights into:

- sales trends
- customer behavior
- product performance

---

## Machine Learning Model

Machine learning models use features stored in the feature store to generate predictions.

Examples include:

- customer churn prediction
- purchase likelihood prediction

Technologies used may include Python ML libraries and model-serving frameworks.

---

# Monitoring and Alerting

The monitoring component tracks pipeline health and data quality.

Metrics monitored include:

- ingestion success rate
- validation error counts
- processing delays

Monitoring tools generate alerts when abnormal patterns occur so engineers can investigate issues quickly.



# Task 2 — Validation and Error Handling

## Task 2.1 — Validation Rules

Validation rules ensure that incoming data is correct, consistent, and usable before entering downstream systems.

### Schema Validation

Schema validation checks whether each field has the correct data type and required structure.

| Field | Type | Required |
|------|------|---------|
| InvoiceNo | String | Yes |
| StockCode | String | Yes |
| Description | String | Yes |
| Quantity | Integer | Yes |
| InvoiceDate | Datetime | Yes |
| UnitPrice | Float | Yes |
| CustomerID | Integer | Yes |
| Country | String | Yes |

Example checks:

- InvoiceNo must not be empty
- Quantity must be an integer
- UnitPrice must be a numeric value
- InvoiceDate must follow datetime format

---

### Value Range Validation

Value validation ensures numbers fall within acceptable ranges.

Examples:

- UnitPrice must be greater than 0
- Quantity must be less than 10,000
- InvoiceDate cannot be in the future

Example invalid record:

| InvoiceNo | Quantity | UnitPrice |
|----------|----------|----------|
| 536365 | 5 | -10 |

This record fails because **UnitPrice is negative**.

---

### Business Rule Validation

Business rules check logical relationships between fields.

Examples:

1. If `InvoiceNo` starts with **C**, it represents a cancellation and the **Quantity must be negative**.
2. The same `StockCode` should map to a consistent `Description`.
3. `CustomerID` should correspond to a valid customer.

Example violation:

| InvoiceNo | Quantity |
|----------|----------|
| C536379 | 6 |

This record is incorrect because **cancellation invoices must have negative quantities**.

---

# Task 2.2 — Error Handling Flow

When validation fails, the pipeline must decide how to handle the problematic record.

### Schema Validation Failure

If a record fails schema validation:

- The record is **rejected**
- It is stored in a **Dead Letter Queue**
- The error is logged for investigation

Example:

InvoiceNo = NULL


Action:

- Record rejected
- Stored in dead letter storage

---

### Value Range Failure

If values are outside acceptable ranges:

- Record is moved to **Quarantine Table**
- Engineers review the issue later

Example:

UnitPrice <= 0


---

### Business Rule Failure

If logical business rules are violated:

- Record is **flagged for review**
- Stored in a **quarantine dataset**

Example:

InvoiceNo starts with "C"

Quantity > 0


This means a cancellation record has positive quantity.

---

### Recovery Process

After errors are corrected:

1. Records are reprocessed from quarantine storage
2. The pipeline runs validation again
3. Correct records move to the transformation stage

Example quarantined record:

| InvoiceNo | StockCode | Quantity | UnitPrice | CustomerID | Status |
|----------|----------|----------|----------|----------|----------|
| C536366 | 85123A | 10 | 5.99 | 17850 | Flagged |

Reason: cancellation invoice with positive quantity.

# Task 3 — Transformation and Storage Design

## Task 3.1 — Transformations

After validation, the transformation stage cleans and enriches the data.

### Cleaning Operations

Common cleaning operations include:

- Removing rows where `CustomerID` is missing
- Removing rows where `UnitPrice ≤ 0`
- Removing duplicate transactions
- Standardizing text formats

Example:

Before cleaning:

| InvoiceNo | CustomerID |
|----------|----------|
| 536365 | NULL |

After cleaning:

Row removed because **CustomerID is missing**.

---

### Derived Columns

New columns are created to help analytics and machine learning.

Example derived column:

LineTotal = Quantity * UnitPrice


Additional useful columns:

- `is_cancellation`
- `year`
- `month`
- `day`

Example:

| Quantity | UnitPrice | LineTotal |
|---------|-----------|-----------|
| 5 | 4.00 | 20.00 |

---

### Customer-Level Aggregations

Customer features are created for analysis.

Examples:

- total_revenue
- total_orders
- average_order_value
- product_diversity
- last_purchase_date
- recency_days

Example:

| CustomerID | Total Orders | Total Revenue |
|------------|--------------|---------------|
| 17850 | 15 | 1200 |

---

### ML Feature Engineering

Features for machine learning models include:

- total_spent_last_90_days
- number_of_orders
- number_of_unique_products
- recency_of_last_purchase

Important rule:

Only **historical data before prediction time** should be used to avoid **data leakage**.

---

# Task 3.2 — Storage Layers

The pipeline stores data in multiple layers.

| Layer | Contents | Format | Update Frequency |
|------|---------|--------|----------------|
| Raw | Original ingested data | Parquet | Continuous |
| Clean | Validated and transformed data | Parquet | Daily |
| Feature | Machine learning features | Parquet | Daily |

### Why Parquet?

Parquet is used because:

- It is a **columnar storage format**
- Provides **high compression**
- Enables **fast analytical queries**

---

# Task 3.3 — Incremental Updates

To avoid processing all data repeatedly, the pipeline uses incremental updates.

### High-Water Mark

The system tracks the latest processed timestamp.

Example:

latest_processed_invoice_date


Only records newer than this timestamp are processed.

---

### Handling Late Arriving Data

Sometimes transactions arrive late.

Solution:

- Reprocess affected partitions
- Update aggregated features

---

### Feature Refresh Strategy

Feature store update strategy:

- Customer features updated **daily**
- Machine learning models retrained **weekly**
